# Design Context — DebtNext.com (dPlat marketing site)

This file is the persistent design brief that informs every UI/UX decision on the dPlat marketing site. It supplements `DESIGN.md` (visual token spec) and `CLAUDE.md` (operating contract) and should be read before any design work.

## Design Context

### Users

**Primary audience:** Buyers at credit originators in regulated industries. Specifically:

- Major US utility providers
- Fortune 100 financial institutions
- Telecom carriers
- Fintech lenders
- Publicly traded debt purchasers
- Healthcare and government recovery operations (secondary)

**Typical buyer profile:** SVP or VP of Recovery / Collections, Director of Operations, Chief Risk Officer, or VP of Credit. Operating budgets in the millions. Owns the recovery vendor network and the financial outcome of it.

**Context when using the site:** Evaluating a recovery management platform against a Q4 RFP cycle, a vendor consolidation initiative, or a regulatory-driven replacement of legacy software. Reading on a managed desktop during work hours, possibly with security-restricted browsers. Comparing dPlat against custom-built solutions, single-vendor agency systems, and legacy recovery software.

**Job to be done:** Replace the patchwork of spreadsheets, vendor portals, and one-off integrations they currently use to manage recovery. Get configurable placement, real-time reporting, and compliance built into one system. Validate that dPlat is enterprise-grade enough to run their portfolio and operationally tight enough to defend in front of compliance and legal.

**Emotional goal:** Confidence the platform is serious. Calm that the buyer is not being sold to. Trust that the regulated framing and disclosures are accurate. Reduce decision risk, not amplify it.

### Brand Personality

**Three words: calm, precise, technical.**

This is software for serious financial work. The personality reads like an operator's tool, not a product pitch. Quiet competence over performance. The interface is the proof — long-form persuasion is not the job.

**Voice attributes (from CLAUDE.md §5):**

- Confident, calm, product-led
- Buyer-focused, not company-focused
- Concrete nouns and specific verbs
- Short paragraphs, varied length, intentional
- Contractions always. Sentence case for headings. Digits for numbers.
- Banned constructions: em dashes, "not X, it's Y", and the banned-phrase list in CLAUDE.md §5

### Aesthetic Direction

**Visual tone:** Mercury-faithful, dark-canvas-first fintech. Premium, controlled, spacious, technical. The page should feel composed before the user reads any copy.

**Source of visual truth:** `DESIGN.md` at the repo root.

**Core token commitments (never deviate):**

- Canvas `#171721`
- Primary action `#5266EB` (one filled pill per band)
- Foreground `#EDEDF3`
- Focus ring `#9CB4E8`
- Inter via `next/font/google` (Arcadia would be preferred if licensed)
- 72-96px desktop section padding · 32-48px mobile
- Pill-shaped primary CTAs · 32px attached-form radii
- Motion is restrained: fades and accordion transitions only, no bounce or parallax

**Single reference site:** [mercury.com](https://mercury.com). Secondary references intentionally avoided to prevent reference dilution.

**Anti-references (do NOT look like):**

1. **TSI corporate sites.** No TSI dark blue, bright blue, Poppins, Calibri, square-corner brand geometry, or TSI logo lockups. dPlat is branded inside the Mercury visual system, not the parent.
2. **Consumer fintech** (Cash App, Robinhood, Chime). Avoid warm gradients, playful illustration, gamified copy, hero CTAs that say "Get started free." The audience is enterprise, not retail.
3. **Generic B2B SaaS templates.** Avoid the Linear/Vercel pastiche — purple-on-white gradient hero, abstract 3D blob, dashboard mockup floating on a tilted gradient block, "AI-powered" copy. If a section could appear unchanged on any B2B SaaS site, rewrite it.
4. **Legacy enterprise collections software** (FICO, TrueAccord legacy UIs, FIS portal aesthetics). Square dense gray dashboards, system-font everything, no whitespace, modal-heavy navigation. dPlat must read as a modern operator tool, not a 2008 enterprise portal.

**AI-generated illustration:** Avoid unless it explicitly passes the dark-fintech aesthetic test in DESIGN.md §11. Cool, crisp, low-noise, atmospheric — never warm consumer gradient or cartoon style.

**Iconography rules:**

- One stroke-weight icon family across the product (currently `1.5px`, currentColor, 24x24). New icons must match.
- One exception: **third-party brand marks** (LinkedIn, GitHub, etc.) render in their canonical form, which is usually `fill` not `stroke`. Don't redraw brand marks to match the in-house stroke style.

### Design Principles

These 5 principles override default UI judgment and apply to every section, every page, every component:

1. **Dark canvas first; light is a deliberate intervention.** The default page experience opens dark. Light sections appear only where they earn the contrast (a deep-dive on a single capability, a long-form reading band). Never as a backdrop "because it's friendlier."

2. **One voltage per band.** Each horizontal band has exactly one filled primary CTA. Secondary actions are ghost, text, or muted treatments. If two filled blues would land near each other, change one. The hierarchy of the page is the hierarchy of the actions.

3. **Whitespace is a design element, not unused space.** Treat 72-96px desktop section padding as load-bearing. Don't crowd cards, product mockups, or text blocks. If a section feels "empty," it probably reads correctly to the buyer.

4. **Product visuals explain; they don't decorate.** Dashboards, workflow diagrams, and screenshots support the adjacent copy. Cool atmospheric imagery is allowed for the hero only. No stock photography, no generic 3D blobs, no decorative illustration that doesn't carry product meaning.

5. **State is explicit and tested.** Every interactive element implements all 7 states from DESIGN.md §7: default, hover, focus-visible, active, disabled, loading, error. Focus rings use `#9CB4E8` at 2px with 2px offset and are never hidden. Reduced motion is honored everywhere.

**Motion:** one engine per job (DESIGN.md §4.7 engine-per-job): GSAP for scroll-scrub and pin cinematics only (lazy, desktop-only, off the `/` eager chunk), Framer for entrances and transitions, CSS for hover, focus, ambient drift, and the 7 states. Consumers import a named primitive from `src/components/motion/`; reveals and live values fail open under reduced motion (final visible state, never stuck at `opacity:0`).

### Accessibility

**Floor:** WCAG 2.2 AA. Tested per CLAUDE.md §11 and verified via `axe-core` in CI (wired in M4).

**Specific commitments:**

- Keyboard navigable end to end, including mobile drawer and accordion
- Focus indicators visible on every interactive element (`#9CB4E8`, 2px outline, 2px offset)
- 44×44px minimum touch targets
- 4.5:1 contrast for body text, 3:1 for large text and UI components
- `prefers-reduced-motion: reduce` disables non-essential motion across the site (hero video hides; accordion snaps; CSS transitions neutralized via the global block in `globals.css`)
- One H1 per page, semantic `<header>` / `<nav>` / `<main>` / `<footer>` landmarks
- Form fields have programmatic labels; error messages are `aria-describedby`-associated
- No reliance on color alone for state communication

### Compliance and disclosure

The dPlat marketing site sits inside a real corporate structure (DebtNext is wholly owned by Transworld Systems Inc.). This affects design in two specific places:

- **Footer ownership line.** "DebtNext is a Transworld Systems Inc. company." renders on every page, persistently, quietly, in `--text-tertiary`.
- **Company page TSI section.** Explicit ownership disclosure with link to TSI. Reviewed by Andrew Budish before any change.

Outside those two surfaces, TSI ownership stays off-screen. No "a TSI company" tag near the dPlat wordmark in the header. No TSI logo lockups. No TSI visual brand language anywhere.

### Performance

The design must hold under these targets (CLAUDE.md §12):

- LCP < 2.5s on 4G
- CLS < 0.1
- INP < 200ms

Concrete commitments that shape design:

- The hero poster image is the LCP target. The hero video layers on top and never owns LCP.
- Below-fold images lazy-load via `next/image`.
- Marketing sections stay Server Components by default. Client components only where state is required (Header scroll state, accordion, mobile drawer, forms, analytics trackers).
- Hero video stays `preload="metadata"` and respects `prefers-reduced-motion`.
