# Phase 14 — UI Design Contract

> Visual and interaction contract for Phase 14 (Text-only page elevation). Generated by gsd-ui-researcher in `--auto` mode; every value below is pre-populated from DESIGN.md, 14-CONTEXT.md (D-01..D-07), 14-RESEARCH.md, the prior 12-UI-SPEC.md, CLAUDE.md, and the shipped Phase 10/11/12 system. No new design decisions or tokens are introduced; this document binds the existing mature system to this phase's 9 target routes plus the two folded form-a11y items (P14-01, P14-02).

**Design-system state: MATURE and CLOSED.** `DESIGN.md` at the repo root is the authoritative token source. This spec does not redefine tokens; it selects and constrains them for this phase. If this document and DESIGN.md ever conflict, DESIGN.md wins and this spec has a bug. The building blocks are fixed: the three archetypes (Console / DataStory / Schematic), the motion primitives (Reveal / Ambient / Marquee / LiveValue), and existing section slots. A new archetype component requires an explicit gap argument raised in planning (stop-list, D-02).

**The operative discipline this phase is RESTRAINT (D-01).** Most target pages get *nothing new* because they already carry section-level scroll-reveal via the legacy `product/motion` barrel. A visual is added only where it advances the page's argument (lift, not decoration). The `14-ARCHETYPE-MAP.md` records the per-page lift/no-lift call as a deliberate decision.

## Typography

Source: DESIGN.md §4.2 / §9.1 fluid type scale (Utopia clamp tokens). This phase adds zero type sizes. The archetype components own all type styling; payloads supply strings only. The subset this phase's surfaces render:

| Role | Size (token) | Weight | Line Height |
|------|--------------|--------|-------------|
| Visual caption / eyebrow / pill text | 12px (`--text-caption`, static) | 420 (UI emphasis) | 1.3 |
| Schematic node labels, DataStory card subs, annotations | 13→14px (`--text-body-sm`) | 400 | 1.5 |
| Body adjacent to visuals, annotation text | 15→16px (`--text-body-md`) | 400 | 1.5 |
| DataStory KPI / card headline numbers | 18→21px (`--text-body-lg` family) | 480 (display emphasis), `tabular-nums` | 1.2 |

Rules bound for this phase:
- Weights: 400 standard, 420 UI emphasis, 480 display emphasis (DESIGN.md §4.2). Never bold-stack inside a visual.
- The 3-weight scale (400/420/480) is DESIGN.md's locked, pre-existing weight system (shipped since Phase 10) — an accepted closed-system exception to the general 2-weight authoring ceiling. This phase introduces no new weights.
- All animated or live numeric values keep `tabular-nums` (no width jitter — `LiveValue` enforces this).
- Sentence case for every payload string: titles, pills, tags, annotations, `ariaSummary`s (CLAUDE.md §5).
- One H1 per page is untouched — visuals are `role="img"` and contribute no headings to the document outline. `/compare`, `/why-dplat`, `/platform/integrations`, and the `/company` set each keep their existing single H1.

## Color

Source: DESIGN.md §4.1. The site's 60/30/10 contract, applied to this phase:

| Role | Value | Usage |
|------|-------|-------|
| Dominant (60%) | `#171721` (`--background` / `--product-canvas`) | Page shell and the canvas inside every archetype frame |
| Secondary (30%) | `#20212d` / `#1e1e2a` (`--card` family) | Elevated surfaces: schematic node fills, DataStory card fills, ambient band backgrounds |
| Accent (10%) | `#5266EB` (`--primary` / `--chart-1`) | The one filled "Request a demo" CTA per band, primary chart series, focus anchors ONLY |
| Destructive / error | `#ef4444` (`--destructive`) | Form-input error border only (DemoForm error state, §8.3). No destructive *actions* exist on this site. |

Accent reserved for: the one filled "Request a demo" pill per band, the primary data series in a chart/schematic, and focus anchors. Never as text on the dark canvas (`--primary` is 3.77:1, fails AA — use `--accent-text-dark #7280F4` for accent text). Never on multiple competing elements in one band (D-03: motion and visuals sit below the CTA in visual weight).

### Accent mapping for new archetypes (no new colors — D-02, CLAUDE.md §3)

Any archetype built this phase draws accents exclusively from the existing chart tokens `--chart-1` (`#5266EB`), `--chart-3` (`#10b981`), `--chart-4` (`#f59e0b`), `--chart-5` (`#06b6d4`). `--chart-2` (`#323649`) is near-canvas and reserved for baselines/grids/muted comparison — not a usable accent (same rule Phase 12 locked, D-11). The `/compare` visual should reserve `--chart-1` (indigo, the brand/dPlat series) for the dPlat side of any comparative contrast and a muted/`--chart-2` baseline for the "typical" or competitor side, so the argument survives grayscale.

### Status color contract (Pitfall 4 — binding)

Every status color is paired with a text label or icon, no exceptions (DESIGN.md §4.1 line 348). Product status tokens where a schematic/data-story surfaces state: `--status-success #5BCB89` (on-time/positive), `--status-warning #FFB86C` (review/pending), `--status-breach #E27676` (negative), `--status-focus #9CB4E8` (live indicator / eyebrow). Every visual must survive grayscale: if rendered without color, the argument must still read from labels, values, and composition. The `ariaSummary` is the screen-reader equivalent of that grayscale story.

## Copywriting Contract

All payload strings and any new caption/label are governed copy under CLAUDE.md §5 and D-04. The new-copy surface this phase is small (~1-3 payload modules × several strings each) but includes the two live governance surfaces: `/compare` comparative claims and the `/company` TSI ownership section.

| Element | Copy |
|---------|------|
| Primary CTA | "Request a demo" — exact label, one filled instance per band, untouched by this phase (CLAUDE.md §4, D-03). Visuals never carry a CTA and never visually compete with the band's CTA. `/demo` gets Ambient/Reveal only precisely so nothing competes with the form's submit CTA (D-01d). |
| Empty state heading | not applicable — payloads are statically typed and always present; an "empty" archetype is a build error, not a runtime state |
| Empty state body | not applicable (same reason) |
| Loading state | Lazy skeleton: `VisualSkeleton` pulse on `--product-canvas`, `rounded-[16px]`, height matched to the resolved visual (see Interaction States) — no text, no spinner |
| Error state (visuals) | Fail-open (FND-01): if a reveal or lazy load fails, content renders fully visible in final state; nothing is ever stuck at `opacity: 0`. No error copy exists because no error UI exists. |
| Error state (DemoForm field) | Existing red-border + adjacent `<p role="alert">` message is unchanged in copy. P14-01 adds the *programmatic association* (`aria-invalid` + `aria-describedby`) so screen readers announce the already-present message. No new error copy. |
| Destructive confirmation | not applicable — no destructive actions on a marketing site |

### Payload / caption copy rules (binding, per-module self-check)

1. No em dashes anywhere. Use `·` as the separator (established console convention, e.g. "3 vendors · 30-day").
2. No "not X, it's Y" constructions (or any variant); state the positive claim directly.
3. No banned phrases (seamless, leverage, robust, unlock, elevate, empower, journey, delve, harness, "digital journeys", etc. — full list CLAUDE.md §5). **AUDIT BL-01:** `compare.ts:99` already contains "digital journeys" (a banned construction) — if `/compare` copy is touched this phase, fix it in-line.
4. Sentence case, contractions, digits for numbers.
5. Every payload module carries `[CLAIMS REVIEW]` / `[COI REVIEW]` header tags — audit-trail markers, non-blocking per Andrew Budish's 2026-06-12 sign-off (D-04), retained for audit; flag both in the PR body.
6. Figures are real-shaped, anonymized, generic; no named clients, no invented testimonials or metrics (CLAUDE.md §15). **The `/compare` comparative claims additionally need a legal gate (punch-list #4) — do not merge comparative-claim copy without it.**
7. Vendor/TSI framing stays agency-network-agnostic ("routes across the originator's existing vendors"); never independence claims (CLAUDE.md §6). The `/company` + `/company/about` TSI ownership sections are live COI surfaces: a visual or heavy motion must not visually bury or reframe them (D-01e).
8. `ariaSummary` is mandatory on every archetype instance — full governed prose that tells the argument in words (see Accessibility).
9. Any `visualLabel` / motion-wrapper `aria-label` fields updated to accurately describe the new visual.

## Interaction States

This phase ships static archetype rendering (no Explorable flagships required — D-01/D-02, distinct from Phase 11). The 7-state matrix from CLAUDE.md §10 applies to the archetypes and the DemoForm as already built; the phase-specific contracts are:

| State | Contract |
|-------|----------|
| Default | Archetype renders fully from its typed payload; all values visible without interaction. `/demo` and the motion-only pages render final content immediately; motion is entrance-only. |
| Loading | Lazy `ssr:false` skeleton. Default `VisualSkeleton` is `min-h-[20rem]`. **`ProductVisualBand` has no height floor (Pitfall 1)** — if a Schematic/DataStory resolves taller than 20rem, reserve the resolved box following the `FLAGSHIP_SKELETON_MIN_H` (`min-h-[44rem]`) precedent in `lazy.tsx` (a dedicated taller skeleton constant or a `min-h` wrapper on the band child). The skeleton must match the resolved box, not approximate it. Verify resolved height on the Vercel preview; CLS gate is < 0.1. |
| Hover | Inherited from archetype internals (CSS engine per DESIGN.md §4.7); no new hover behavior added. |
| Focus-visible (general interactive elements) | DESIGN.md §8: 2px `#9CB4E8` outline, 2px offset. Applies to links, buttons, tabs, accordion triggers — unchanged, already compliant. |
| Focus-visible (form inputs — P14-02) | DESIGN.md §8.3 Input states + `shadows.focus` token: border shifts to `#5266EB` (`--primary`) **plus a 3px `#9CB4E8` ring** (`0 0 0 3px rgba(156, 180, 232, 0.35)`, the `shadows.focus` token). **This is intentional and on-spec for inputs** — the general 2px-outline rule does not govern text inputs. | Active / filled | Preserve text contrast and border clarity (§8.3). No new transitions. |
| Disabled | Lower opacity, no pointer events, disabled cursor (§8.3). No new disabled controls added. |

| Reduced motion | **Full content parity, zero motion** (FND-01, Phase 11/12 contract): every value, label, and composition a motion-enabled user sees is present and final for a `prefers-reduced-motion: reduce` user. `RevealSection` returns children untouched; `AmbientField` collapses to a static bloom; `LiveValue` lands on the displayed value immediately. Asserted per touched route by `14-page-elevation.spec.ts` + the existing `reduced-motion.spec.ts` / `reveal-fail-open.spec.ts` (both iterate `VISUAL_ROUTES`, which already lists all 9 targets). |

### Motion contract

Engine-per-job (DESIGN.md §4.7): all motion this phase comes from the motion barrel via the archetypes or `RevealSection` / `AmbientField` — Framer for entrances/reveals, CSS for ambient/hover. **No GSAP** (no scroll-scrub/pin surfaces in scope; GSAP stays the single dynamic-import owner in `HeroCinematicController`, never touched). Pages import only the lazy `*Visual` wrappers plus payload modules, or `RevealSection` / `AmbientField`. Never import `Console.tsx` / `DataStory.tsx` / `Schematic.tsx` directly (Pitfall 5 — eager-import TBT blowout against the 300ms content-route bar).

**Do NOT double-wrap** a section whose inner component already animates via `product/motion` (`CardGrid`, `BulletList`, `CompareMatrix`, `ComparisonTable`, `ProcessStrip`, `IntegrationTable`, `LeadershipTable`, `IntegrationStrip`) in a `RevealSection` — that stacks two entrances and reads as jank (Pitfall 3). Only wrap sections that do not self-animate (`ProseSection`, raw `SectionContainer` blocks), which is exactly what about/leadership/careers/contact already do.

### P14-02 resolution (documented as intentional — the design decision this spec locks)

The current form-input focus treatment (`focus:ring-3 focus:ring-[var(--focus)]/35` + `focus:border-[var(--primary)]`) is **on-spec for inputs** and is hereby documented as the intentional §8.3 input-focus pattern, NOT normalized to the general 2px outline:
- DESIGN.md §8.3 Input states specify: focus-visible border `#5266EB` + a focus ring / background tint.
- The `shadows.focus` token (DESIGN.md line 133) is literally `0 0 0 3px rgba(156, 180, 232, 0.35)` — a **3px** `#9CB4E8` ring.
- Normalizing to the 2px outline would regress the `AttachedForm` pill aesthetic and contradict the token.

**Action for the planner/executor:** add a DESIGN.md §8.3 note (in the same commit as any code touched) recording that the 3px `#9CB4E8` input ring is the sanctioned input-focus pattern, citing `shadows.focus`. The one genuine inconsistency to fix is `AttachedForm.tsx:55`, which has border-only and **no ring** — bring it in line with the §8.3 ring. `HomepageHero.tsx:236` is a **Phase-13/15 firewall file**: if any P14-02 change must reach it, land it on its own track and raise it — do NOT touch it silently (D-05b).

## Accessibility Contract

- Every archetype instance carries `role="img"` + a complete `ariaSummary` (Phase 11/12 pattern) — governed prose telling the argument in words, doubling as the per-route distinctness assertion hook in the spec.
- Status colors always label-paired (text or icon beside color); the grayscale-survivability test is the bar (Pitfall 4).
- WCAG 2.2 AA contrast on every pairing; body text on dark uses `#EDEDF3` / `#C3C3CC` hierarchy; `--primary` never as text on dark.
- 44px touch floor on every interactive element (CLAUDE.md §11). DemoForm inputs touched by P14-01/P14-02 stay ≥ 44px; visible focus indicator uses `#9CB4E8`. `axe-routes.spec.ts` + `touch-targets.spec.ts` already iterate all 23 `VISUAL_ROUTES` (AUDIT FIX-02) and must stay green.
- **P14-01 (binding):** DemoForm error fields expose `aria-invalid="true"` and `aria-describedby="{id}-error"` when errored. Recommended implementation: wire it in `Field` (where `id` and `error` are both in scope) or extend `inputClasses(error, id)` so all 6 inputs + 2 selects + 1 textarea are fixed in one edit. New/extended a11y assertion asserts both attributes on each errored field after triggering validation.
- One H1 per page unchanged; visuals contribute nothing to the heading outline.
- `AmbientField` is `aria-hidden`, seeded (no hydration mismatch), and collapses to a static bloom under reduced motion — never an accessibility or CLS liability.

## Restraint / Lift Contract (D-01 — the reason this phase is different)

This is a verifiable gate, not a vibe (Pitfall 2):

1. **Map gate first:** `14-ARCHETYPE-MAP.md` (Phase 12 D-08 precedent) is produced before any page work and records, per target route, the explicit lift/no-lift call, and for each lift: the archetype, the slot, the payload intent, and the per-route distinctness strings. A no-lift call is a deliberate decision, recorded — not an omission. Connor's map approval is the reviewer gate.
2. **The bar for a visual:** the map states *what argument the visual advances that the prose/table does not*. If the answer is "it looks nice," the correct output is Reveal/Ambient or nothing. Blanket coverage violates D-01.
3. **Starting hypotheses (planner confirms/overrides in the map):**
   - `/compare` → **DataStory or Schematic** (time-to-production / continuous-production proof, or network-agnostic routing) that makes the comparative point without duplicating the market-map table. Live `[CLAIMS REVIEW]` + legal + `[COI REVIEW]` surface. (D-01a, PAGEVIS-01)
   - `/platform/integrations` → **Schematic (system/network map)**, system-consistent with the `PlatformSystemMap` precedent; no CTA competition. (D-01c, PAGEVIS-04)
   - `/why-dplat` → **DataStory (proof)** on the ProofBand story, OR Reveal-only if a figure would duplicate the already-animating argument bands. (D-01b, PAGEVIS-02)
   - `/demo` → **Ambient/Reveal only**, never an archetype and never competing with the submit CTA; pair with P14-01/P14-02. (D-01d, PAGEVIS-04)
   - `/company` set + `/resources` → **Reveal-only or nothing** (sections already animate; TSI/COI section stays unobscured). Record the no-lift call explicitly. (D-01e, PAGEVIS-03)
4. **Spec gate second:** `14-page-elevation.spec.ts` asserts per-route argument-unique strings (e.g. an integrations-only node label, a compare-only annotation) on load AND under reduced motion. "`role=img` exists" alone is insufficient — it cannot catch a copy-pasted or decorative payload (Pitfall 7).
