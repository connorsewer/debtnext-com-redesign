# Phase 05.2 Plan 01: AVIF poster swap (Summary)

Three commits shipped this plan. The dominant win was the AVIF swap (`1ee187b`); two follow-up commits chased remaining diagnostic leads from the LHCI re-runs but did not move the simulated-LCP needle. The 2,300 ms gate is still red. Phase 05.2 is `PARTIAL`. Phase 05.1 Plan 03 Tasks 2 + 3 stay deferred until a future phase or gate renegotiation closes HERO-04 fully.

## What shipped

### Commit 1: `feat(05.2): swap hero poster to AVIF to close HERO-04 LCP gate` (1ee187b)

**The swap:**
- `public/hero/homepage-hero-start.avif` created (112 KB, libsvtav1 CRF 30 preset 5; 56% of 200 KB HERO-03 budget)
- `public/hero/homepage-hero-start.png` deleted (was 2.55 MB raw)
- `src/content/homepage-hero.ts:55` `startFrame` repointed to `.avif`; comment block refreshed to document both consumers
- `scripts/encode-hero-poster.sh` new + executable: extracts first frame from `public/hero/homepage-hero-720p.mp4`, re-encodes via ffmpeg libsvtav1, refuses to overwrite if output exceeds 200 KB
- `scripts/check-hero-assets.sh` comment updated to drop the obsolete "raw PNG is the AVIF source" note
- `.gitignore` appended with `/.lighthouseci/`
- `.planning/phases/05.2-swap-hero-poster-to-avif/05.2-CONTEXT.md` + `05.2-01-PLAN.md` created
- `.planning/ROADMAP.md` + `.planning/STATE.md` + `HANDOFF.md` updated

**Result on Vercel preview LHCI Case C run:** simulated median LCP dropped from **16,411 ms to 3,488 ms** (12,923 ms reduction; 4.7x improvement on this single commit alone). Gate still red by 1,188 ms.

### Commit 2: `fix(05.2): drop redundant <video poster> to free mobile bandwidth` (84f20dd)

**The fix:** removed `poster={heroCinematic.media.startFrame}` from the `<video>` at `HomepageHero.tsx:193`. Layer 1 (`<Image src={startFrame} preload fetchPriority="high">` at `HomepageHero.tsx:170-181`) already paints the same start frame full-cover under the video on every viewport; the `<video>` itself starts at `style={{ opacity: 0 }}` and only fades in via GSAP scrub. The poster attribute was triggering a redundant 110 KB AVIF fetch on mobile via SSR HTML before hydration removed the element.

**Result:** simulated median LCP 3,488 ms to 3,648 ms (statistical noise within the run-to-run variance; the 110 KB was no longer the bottleneck after AVIF shrunk the asset). The change still removes real waste from the page (one round-trip + bandwidth contention) and is visually neutral, so it shipped.

### Commit 3: `fix(05.2): pin Inter font-display to "optional" to stop H1 LCP swap` (8920088)

**The hypothesis:** the 2,204 ms gap between FCP (1,635 ms) and LCP (3,839 ms) on the post-poster-removal preview pointed at the Inter font swap. With `display: "swap"` the H1 first paints in next/font's metrics-matched fallback at FCP, then re-paints when Inter arrives, which the browser registers as a new LCP candidate. Switching to `display: "optional"` gives the browser ~100 ms to fetch Inter; if it does not arrive in that window the fallback locks in for the session, no swap, no second LCP candidate.

**The fix:** `src/app/layout.tsx` Inter config changed from `display: "swap"` to `display: "optional"` with an inline comment explaining the LCP rationale.

**Result:** simulated median LCP 3,648 ms to 3,869 ms (regression within noise; the hypothesis was wrong). The LCP-FCP gap actually grew (2.1-2.9 s) on the post-font-display preview, ruling out font swap as the active bottleneck. The change still has a defensible long-term rationale (no font-swap layout risk) so it shipped, but it did not close the gate.

## Why the gate stays red

`docs/m5-phase-5-lhci-run.md` carries the full diagnosis. Short version: LCP element is the H1; observed (unthrottled) LCP equals FCP equals 1,254 ms (no actual render-delay phase). The simulated LCP at 3,869 ms is LHCI's projection of how the resource graph + CPU graph would perform on Case C throttling. The remaining levers are not asset-level:

- JS critical path: 165 KB of `_next/static/chunks/` on `/`, 78 KB of which is unused. GSAP + ScrollTrigger are eagerly imported in `HomepageHero.tsx` but only the desktop cinematic uses them.
- General Sans `<link rel=preload>` ships in `<head>` from `layout.tsx` (HERO-02). Only the desktop Wordmark uses it; mobile pays 47 KB regardless.
- Vercel preview cold-lambda TTFB modeling under simulated throttling adds latency that does not reflect production CDN behavior.
