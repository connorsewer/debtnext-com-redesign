# Phase 05.2: Swap hero poster to AVIF (Context)

**Gathered:** 2026-05-21
**Status:** Executed (no ceremony; diagnosis was locked by Phase 05.1 Plan 03 Task 1 LHCI failure)

## Phase Boundary

Phase 05.1 closed the mobile-video-free contract structurally (Plans 01 + 02 sealed D-04 and added regression nets), then Plan 03 Task 1 re-ran the LHCI Case C gate against the Vercel preview. Result: median LCP 16,411 ms vs 2,300 ms gate. Per-run 16,411 / 16,365 / 15,479 ms. The H1 was the LCP element; the network waterfall surfaced a 2.55 MB raw PNG fetch (`/hero/homepage-hero-start.png`) that was holding the H1 paint back at 1.6 Mbps throttled bandwidth.

The PNG was wired in two places at `src/content/homepage-hero.ts:55`:
1. `<Image src={startFrame} preload fetchPriority="high">` at HomepageHero.tsx:171: goes through `/_next/image` and gets transcoded to AVIF on `Accept: image/avif` (HERO-03 contract). Not the problem.
2. `<video poster={startFrame}>` at HomepageHero.tsx:193: fetches the raw URL directly, bypassing `/_next/image`. Even though the `<video>` element itself is React-gated behind `!isMobile`, SSR renders it (`isMobile` defaults to `false` server-side) so the browser starts the 2.55 MB PNG fetch before hydration's `matchMedia` check removes the element. At 1.6 Mbps that fetch alone takes ~13 seconds, contending for the same bandwidth window the H1 needs.

This is the A5 risk recorded in `05.1-CONTEXT.md`. Phase 05.2 closes it with a one-asset swap.

In-scope for Phase 5.2:
- Encode `public/hero/homepage-hero-start.png` (2.55 MB) to `public/hero/homepage-hero-start.avif` (112 KB at libsvtav1 CRF 30)
- Repoint `startFrame` in `src/content/homepage-hero.ts:55` to the `.avif` path
- Delete the raw PNG (regenerable from `homepage-hero-720p.mp4` via the new encode script)
- Land `scripts/encode-hero-poster.sh` so the AVIF can be re-derived on cinematic asset changes
- Touch up the comment in `scripts/check-hero-assets.sh` (no PNG byte budget needed; the AVIF poster budget is enforced by `tests/hero/poster-avif-negotiation.spec.ts`)
- Add `.lighthouseci/` to `.gitignore` (avoids checking in local LHCI run artifacts)
- Update `.planning/ROADMAP.md` + `.planning/STATE.md` to reflect 5.2 execution
- Phase 05.1 Plan 03 Task 1 resumes against the post-5.2 Vercel preview (separate phase, separate commit)

Out of scope for 5.2:
- Server-side User-Agent phone detection (the AVIF reduction is enough on its own; the post-hydration `<video>` removal stays as belt-and-suspenders)
- Conditional `poster` attribute (no behavior change required once the static asset is 23× smaller)
- `<picture>` wrapping of `<video>` (not valid HTML for the poster path)
- LHCI re-run (lives in Phase 05.1 Plan 03 Task 1 per D-08 close-out plan)
- HERO-01..04 traceability flips (lives in Phase 05.1 Plan 03 Task 3, which closes Phases 5 + 5.1 + 5.2 together via one atomic doc-sync commit)

## Implementation Decisions

### D-09: Single static AVIF, no conditional render

Land one `homepage-hero-start.avif` and point both consumers at it. The `<Image>` consumer still gets transcoded sizes through `/_next/image` (Next/Image accepts AVIF as a source and re-encodes per viewport, so HERO-03's existing contract continues to hold). The `<video poster>` consumer fetches the static 112 KB asset directly. No SSR user-agent detection, no `<picture>` wrapping. Smallest blast radius that closes the LCP gate.

### D-10: Delete the raw PNG; MP4 ladder is the regenerable source

The PNG was the AVIF source in HERO-03. Once the static AVIF ships, the PNG has no production consumer and no need to live in `public/hero/`. The first frame of `homepage-hero-720p.mp4` is the same image. `scripts/encode-hero-poster.sh` extracts that frame and re-encodes if the cinematic asset ever changes.

### D-11: libsvtav1 CRF 30 preset 5

The ffmpeg build available locally (Homebrew) ships `libsvtav1` as the AV1 encoder. At CRF 30 / preset 5 the 1536×1024 poster encodes to 112 KB, which is 56% of the 200 KB HERO-03 poster budget. The script refuses to overwrite the output if it ever blows past that budget; a future tuning can raise CRF.

### D-12: One atomic Phase 05.2 commit

Roadmap insertion artifacts, AVIF binary, source code edit, PNG removal, regen script, planning docs, `.gitignore`, and HANDOFF.md update ship in one commit. The per-commit docs rule (`feedback_docs_in_sync.md`) is honored end-to-end. The next commit (LHCI re-run docs) belongs to Phase 05.1 Plan 03, not 5.2.

## Threat Surface

- T-05.2-01: AVIF poster fetched by a browser without AVIF support (e.g., very old Safari/Edge) leaves `<video>` without a paint frame until the video itself starts buffering. **Mitigation:** AVIF in `<video poster>` is supported across all evergreen browsers (Chrome 85+, Firefox 113+, Safari 16+, Edge 90+); pre-2023 holdouts simply paint the vignette + headline directly, with no broken UX, just no poster preview. Accepted.
- T-05.2-02: AVIF encoder updates could shift output size. **Mitigation:** `scripts/encode-hero-poster.sh` fails fast if the output exceeds 200 KB, matching the existing `poster-avif-negotiation.spec.ts` contract.
