# Phase 05.1 Plan 02: Mobile-video-free regression net + asset size budget guard — Summary

Two regression nets land in this plan plus one Plan 01 bug fix that the first net immediately surfaced. The Playwright spec watches the network at 412x823 and asserts zero video requests fire. The shell script enforces per-file MP4 size budgets and is wired into CI before the LHCI step. The bug fix bounds the 720p `<source>` so no source qualifies at <768px (Plan 01 had left it unbounded on a wrong reading of the HTML source-selection algorithm).

## What shipped

### Commit 1 — `fix(05.1-02): bound 720p <source> with (min-width: 1440px) to seal D-04` (084de36)

**The defect (Plan 01 carry-over):** Plan 05.1-01 SUMMARY's "Phone Render Contract" claimed that at 412x823, the browser would land on `NETWORK_NO_SOURCE` because the bounded 360p/540p sources fail their `(min-width: 768px)` predicate. The 720p source had no `media` attribute, with a comment saying it "applies only above 1440px implicitly because narrower viewports match the bounded sources first." This is wrong per the HTML5 source-selection algorithm: a `<source>` with no `media` attribute always passes the media check, so when the bounded predecessors fail, the unbounded 720p qualifies and the browser picks it. Plan 02's new mobile-video-free spec caught this on its first run by observing `http://localhost:3000/hero/homepage-hero-720p.mp4` in the request log at 412x823.

**The fix:**
- `src/content/homepage-hero.ts` — 3rd `<source>` gets `media: "(min-width: 1440px)"`. Comment block updated to document the corrected behavior (every source bounded; no unbounded fallback).
- `tests/hero/source-ladder.spec.ts` — `expected[2].media` flips from `null` to `"(min-width: 1440px)"`. The null-branch in the assertion loop is removed (every variant now has a media attribute).

This is Rule 1 (bug) + Rule 2 (missing critical functionality — D-04's mobile-video-free contract) per the deviation rules. Auto-fixed inline. Plan 01 SUMMARY's "Phone Render Contract" section should be considered superseded by this commit's behavior.

### Commit 2 — `feat(05.1-02): add mobile-video-free Playwright regression net (D-06)` (10c562c)

**New file:** `tests/responsive/hero-mobile-video-free.spec.ts` (47 lines).

Verbatim from RESEARCH §E recommended implementation. The spec:
- Sets viewport to 412x823 (matches LHCI mobile profile from `lighthouserc.json` screenEmulation exactly).
- Registers `page.on("request", ...)` BEFORE `page.goto("/")` so the observer is live for every request including the initial HTML.
- Filters requests with `/\.(mp4|webm|m4v|mov)(\?|$)/i` — case-insensitive, with `(\?|$)` boundary so query-string-bearing URLs match while substrings (e.g., `imp4ct.png`) don't.
- Scrolls to bottom, waits, scrolls back to top, waits — defense against future lazy-loaded video.
- Asserts `videoRequests.toHaveLength(0)` with a diagnostic that prints the offending URLs.

Why `page.on('request')` not `page.route()`: this is a read-only observer that catches anything the browser fetches regardless of fetch path (HTML `<source>`, `<link rel=preload>`, service worker, dynamic JS). `page.route()` intercepts and lets you mock — not what we want for "fail if X happens."

### Commit 3 — `feat(05.1-02): add hero asset size budget guard + wire into CI (D-07)` (fc4712d)

**New file:** `scripts/check-hero-assets.sh` (53 lines, executable).

Verbatim from RESEARCH §F recommended implementation. The script:
- Declares 3 MP4 budgets in a bash array: `homepage-hero-720p.mp4:10485760`, `homepage-hero-540p.mp4:6291456`, `homepage-hero-360p.mp4:3145728`.
- Walks each entry, checks the file exists, reads its byte size with a macOS (`stat -f %z`) / Linux (`stat -c %s`) fallback.
- Prints `PASS  <file>: <bytes> bytes (<pct>% of budget)` on success per file, or `FAIL  <file>: <bytes> bytes > <budget> bytes budget` on failure.
- Aggregates a single `FAIL` exit code; exits 0 if all 3 pass, 1 if any fail or are missing.
- Uses `set -euo pipefail` for defensive shell.

Current sizes (verified at HEAD):
```
PASS  homepage-hero-720p.mp4: 9769154 bytes (93% of budget)
PASS  homepage-hero-540p.mp4: 4804833 bytes (76% of budget)
PASS  homepage-hero-360p.mp4: 1994588 bytes (63% of budget)
```

**3-budget shape rationale (RESEARCH §F):** Researcher recommended dropping the user-locked AVIF (200 KB) and PNG (300 KB) budgets from D-07:
- The AVIF is generated on-demand by Sharp from the raw PNG; there's no static AVIF file in `public/hero/`. The 200 KB AVIF contract is already enforced by `tests/hero/poster-avif-negotiation.spec.ts` (line ~47, `buffer.byteLength < 200_000`).
- The raw PNG (`homepage-hero-start.png`, 2.55 MB) is the AVIF source. Next/Image transforms it for every viewport; clients never see the raw PNG. A 300 KB raw-size budget would force AVIF degradation. Out of scope for 5.1.

Plan 02 implements the 3-budget shape. If a future need emerges to size-check additional assets, the BUDGETS array extends trivially.

**CI wiring:** `.github/workflows/perf.yml` — added a `Verify hero asset budgets` step between `Build` and `Run Lighthouse CI`. A budget violation fails the workflow before the 90-second LHCI run starts. Diff:

```yaml
       - name: Build
         run: npm run build
+      - name: Verify hero asset budgets
+        run: bash scripts/check-hero-assets.sh
       - name: Run Lighthouse CI
         run: npx --no-install lhci autorun
```

## Deviations from Plan

### [Rule 1 + Rule 2] Plan 05.1-01 720p source was unbounded; sealed in commit 084de36

- **Found during:** Task 1 verification (`npx playwright test tests/responsive/hero-mobile-video-free.spec.ts`).
- **Issue:** Plan 01 SUMMARY claimed the bounded 360p/540p sources alone would make phones match zero `<source>` children, with the 720p left unbounded. The new spec immediately surfaced a real network request: `http://localhost:3000/hero/homepage-hero-720p.mp4` fired at 412x823. Root cause: a `<source>` with no `media` attribute passes the media check per the HTML source-selection algorithm; "no media attribute" does not mean "skip this source."
- **Fix:** Added `media: "(min-width: 1440px)"` to the 720p entry in `src/content/homepage-hero.ts`; updated comment block to document the corrected reasoning. Updated `tests/hero/source-ladder.spec.ts` to assert the new media attribute on the 3rd source.
- **Files modified:** `src/content/homepage-hero.ts`, `tests/hero/source-ladder.spec.ts`.
- **Commit:** 084de36.
- **Why auto-fixed:** Rule 1 (the source ladder didn't behave as documented in Plan 01 SUMMARY) + Rule 2 (D-04's mobile-video-free guarantee is a correctness requirement, not a feature). The fix is one media attribute on one line plus a comment refresh.
- **Plan 01 SUMMARY impact:** the "Phone Render Contract" section in `05.1-01-SUMMARY.md` is superseded. The corrected contract is: every source bounded at `(min-width: 768px)` or higher, so phones below 768px match nothing and the browser starts zero downloads (verified at the network layer by the Plan 02 spec).

### [Pedantic] Acceptance criteria say "exactly 1" for budget grep counts; verbatim script has 2 occurrences

- **Issue:** Plan 02 Task 2 acceptance criteria say `grep -c "10485760" scripts/check-hero-assets.sh` returns exactly `1` (likewise for `6291456` and `3145728`). The verbatim RESEARCH §F template — which the plan instructs to copy without paraphrasing — contains each budget number twice: once in the comment block on line 18 (`# reports: 10 MB = 10485760, 6 MB = 6291456, 3 MB = 3145728.`) and once in the `BUDGETS` array. The actual count is `2` per number.
- **Disposition:** Implemented the verbatim template (the script works correctly: 3 PASS lines, all checks pass). This is a minor inconsistency between the criterion and the template in the plan itself, not a defect in the implementation.
