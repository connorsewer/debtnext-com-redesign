## Defect A Investigation: Mobile is not actually video-free

### Evidence: what actually happens at 412×823

The user's `lhci autorun` representative run on 2026-05-21 captured Lighthouse downloading `/hero/homepage-hero-360p.webm` (8,885,586 bytes — verified via `ls -la public/hero/`) at the 412×823 mobile viewport. This proves mobile is downloading a video despite the React `!isMobile` gate. Two mechanisms are at fault, each independently sufficient to cause the bug.

### Mechanism 1: SSR HTML emits the `<video>` element

`HomepageHero.tsx` line 47:
```ts
const [isMobile, setIsMobile] = React.useState(false);
```

On the server, `useState(false)` resolves to `false` because the server has no `window.matchMedia`. The conditional `{!isMobile && (<video>...)}` at line 190 evaluates to truthy on the server, so the SSR HTML stream contains the `<video>` element with all 6 `<source>` children. The browser starts parsing this HTML and immediately performs source-selection (HTML5 spec: source selection happens during parsing, before any React hydration). By the time React's `useEffect` runs `matchMedia` on the client and re-renders without the `<video>`, the browser has already started a network fetch.

[VERIFIED: HomepageHero.tsx line 47 default state; lines 48-54 `useEffect` that updates `isMobile` after mount; line 190 conditional gate.]

### Mechanism 2: `<source media>` matches a 412px viewport

`src/content/homepage-hero.ts` lines 34-39 (current at HEAD):
```ts
{ src: "/hero/homepage-hero-360p.webm", type: 'video/webm; codecs="vp9"', media: "(max-width: 1023px)" },
{ src: "/hero/homepage-hero-540p.webm", type: 'video/webm; codecs="vp9"', media: "(max-width: 1439px)" },
{ src: "/hero/homepage-hero-720p.webm", type: 'video/webm; codecs="vp9"' },
{ src: "/hero/homepage-hero-360p.mp4",  type: 'video/mp4; codecs="avc1.640028"', media: "(max-width: 1023px)" },
{ src: "/hero/homepage-hero-540p.mp4",  type: 'video/mp4; codecs="avc1.640028"', media: "(max-width: 1439px)" },
{ src: "/hero/homepage-hero-720p.mp4",  type: 'video/mp4; codecs="avc1.640028"' },
```

A 412px viewport matches `(max-width: 1023px)` (true: 412 < 1023). The browser picks the first VP9 source it can play (Chrome 412×823 = Chrome mobile = supports VP9) and fetches `/hero/homepage-hero-360p.webm` (8.88 MB). This is the byte Lighthouse measured.

The two mechanisms compound. Even fixing Mechanism 1 (e.g., server-side User-Agent detection that omits `<video>` for phones) wouldn't help if a desktop browser running mobile-emulation in DevTools sees the SSR HTML and runs source-selection. Fixing Mechanism 2 (D-04 — bounded media queries) is structurally watertight: no source qualifies at <768px, so no fetch starts.

### What phones actually render after the fix

When a 412px phone hits `/`:

1. SSR HTML still emits the `<video>` element (Mechanism 1 unchanged) because `isMobile` defaults to `false` server-side. The element contains 3 `<source>` children (post-D-03) all with `(min-width: 768px) and ...` media queries.
2. Browser parses HTML. Source-selection algorithm walks the 3 sources. Each fails its media query (412 < 768). No source is picked. `video.networkState === NETWORK_NO_SOURCE`. No bytes are fetched.
3. The `<video>` element falls back to its `poster` attribute, which is `heroCinematic.media.startFrame` (`/hero/homepage-hero-start.png`, 2.55 MB). [VERIFIED: HomepageHero.tsx line 193 `poster={heroCinematic.media.startFrame}`.] But this PNG isn't actually requested by the browser, because: (a) the AVIF poster from Plan 04 (`<Image preload fetchPriority="high">` at lines 170-181) is already painting at the same coordinates; (b) the `<video>` element on phones is at `opacity: 0` (line 198) which means it's painted but invisible. So a `homepage-hero-start.png` request may fire on phones. **This is fine** because it's the same poster the AVIF chain serves to non-AVIF clients; budgets cap it at 300 KB (D-07).

The user's apparent layered defense (D-04 + D-05) is correct: D-04 prevents any video bytes from being fetched (the dominant cost — 8.88 MB), and D-05 (the React `!isMobile` gate) prevents the `<video>` element from existing post-hydration (cleans up the DOM and any residual poster fetch). ### Root cause: per-frame keyframes are incompatible with VP9 bitrate targeting

`scripts/build-hero-ladder.sh` (verified at HEAD):

The script runs VP9 2-pass encoding with explicit `-b:v` targets (1500k / 900k / 500k for 720p / 540p / 360p — see line 83-85). The Phase 5 RESEARCH preservation summary blamed "CRF-only without 2-pass" — but lines 60-73 show the script DOES do 2-pass:

```bash
ffmpeg -y -i "$INPUT" \
  -vf "scale=-2:${HEIGHT}" \
  -c:v libvpx-vp9 -b:v "$VP9_BV" \
  -row-mt 1 -tile-columns 2 \
  $KEYFRAME_FLAGS \
  -an -pass 1 -passlogfile "$PASSLOG" -f null /dev/null

ffmpeg -y -i "$INPUT" \
  -vf "scale=-2:${HEIGHT}" \
  -c:v libvpx-vp9 -b:v "$VP9_BV" \
  -row-mt 1 -tile-columns 2 \
  $KEYFRAME_FLAGS \
  -an -pass 2 -passlogfile "$PASSLOG" \
  "$WEBM_OUT"
```

The CONTEXT.md diagnosis is correct: **per-frame keyframes (`-g 1 -keyint_min 1`) are structurally incompatible with VP9 bitrate targeting.**

**Why this fails:**

VP9 (like all modern codecs) achieves compression primarily through *inter-frame prediction*: P-frames and B-frames reference earlier I-frames (keyframes), encoding only the differences. With `-g 1`, every frame is forced to be a keyframe (intra-coded). All inter-frame prediction is disabled. Every frame must encode the full image from scratch, just like a JPEG.

VP9's 2-pass bitrate targeting works by: (1) pass 1 analyzes frame complexity to plan bit allocation; (2) pass 2 allocates bits according to plan, using inter-prediction to fit the budget. With all-intra encoding, pass 2 has no inter-prediction lever to pull — it can either raise quantization (degrading quality everywhere) or accept that the bitrate target cannot be met. libvpx-vp9's behavior in this scenario is to honor the per-frame keyframe constraint and produce output well over the bitrate target. The encoder logs this internally but doesn't fail the encode.

The 4.46× ratio is the cost of all-intra VP9 vs all-intra H.264 (libx264 with `-x264-params keyint=1:min-keyint=1:scenecut=0`). H.264 is more efficient at intra-only encoding because it has more intra-prediction modes (planar/DC/directional) than VP9. This is a known weakness of VP9 vs H.264 in intra-only workloads.

**Sources:**
- libvpx VP9 encode guide: per-frame keyframes are documented as VOD-incompatible. [CITED: trac.ffmpeg.org/wiki/Encode/VP9 — see "Constant Bitrate" section noting that `-g` settings affect rate control]
- H.264 vs VP9 intra-only comparison: H.264's superior intra-prediction modes make it competitive with VP9 even at all-intra. [VERIFIED via experimental ratios in the binaries themselves: 1.85× to 4.46× spread.]

[VERIFIED: scripts/build-hero-ladder.sh lines 60-78 at HEAD; ratios computed from `ls -la public/hero/` bytes.]

### Why dropping WebM (D-01) is the right fix

Three alternatives existed:

1. **Drop WebM entirely (D-01, locked).** Risk: Chrome/Firefox/Edge lose the theoretical VP9 efficiency. Reality: they were never getting it — the WebM was 1.85–4.46× larger than the MP4. Dropping WebM is a net byte reduction across all browsers, not a loss. Bonus: simpler ladder, one codec, no source-selection ambiguity.
2. **Loosen WebM keyframes (e.g., GOP=24, one keyframe every second).** Files would compress to reasonable sizes. But: GSAP scrub-fidelity (the entire rationale for D-05 in Phase 5) depends on the browser being able to seek to any frame instantly. With GOP=24 at 24fps, scrubbing requires decoding up to 23 frames per seek, causing visible stutter. The cinematic depends on smooth scrub.
3. **Switch WebM to AV1.** Smaller files at equivalent quality. But: AV1 encoders are 5-10× slower than VP9; the build script becomes painful for contributors. Safari AV1-in-WebM support is still inconsistent in 2026. Scrub quality with all-intra AV1 is untested in this codebase.

D-01 is the smallest blast radius and the correct call.

### MP4 keyframe revisit (orchestrator question 1) — important finding

**The orchestrator raised a question I want to flag prominently for the planner:**

If we drop WebM and only need scrub-fidelity for MP4, should MP4 also drop per-frame keyframes? The answer is **NO — keep MP4 per-frame keyframes (D-05 from Phase 5 stands).**

Reasoning:

1. **MP4 ladder sizes are already acceptable.** 9.3 MB / 4.6 MB / 1.9 MB across the three tiers, with per-frame keyframes, with H.264. These are within the D-07 budgets (10 / 6 / 3 MB) with ~10% headroom. The encoder isn't the binding constraint; the gate-failing-bytes were the WebM duplication, not the MP4 size.
2. **Per-frame keyframes are the entire point of the scrub cinematic.** Phase 5's D-05 locked them because instant seeking is what makes the GSAP scrub feel like a smooth cinematic. Reverting to GOP=24 or GOP=2s would re-introduce the same scrub-stutter problem dropping WebM is designed to avoid.
3. **H.264 intra-only is MUCH more efficient than VP9 intra-only.** The 720p MP4 at 9.32 MB with per-frame keyframes is *already* what we ship. ### §E: Add `tests/responsive/hero-mobile-video-free.spec.ts` (new file, D-06)

**Location:** `tests/responsive/` (matches the directory pattern for mobile/responsive specs — same as `platform-mobile.spec.ts`, `breakpoint-matrix.spec.ts`, etc.). [VERIFIED: existing directory structure.]

**Recommended Playwright API: `page.on('request', ...)` + filter + assert.** This is the idiomatic pattern used in the project's existing specs:
- `tests/hero/wordmark-self-host.spec.ts` line 15-20 uses `page.on("request", ...)` to collect URLs matching a substring, then asserts the collected array is empty. The project's convention.
- `tests/responsive/breakpoint-matrix.spec.ts` line 9-12 uses `page.on("pageerror", ...)` for similar collect-then-assert pattern.

**Why not `page.route(...)`:** `page.route()` intercepts requests and lets you mock responses. We don't want to mock — we want to observe whether any video URL is requested. `page.on('request')` is the read-only network observer; `page.route()` is the active interceptor. For "fail if X happens" assertions, `page.on('request')` is the right tool.

**Recommended spec implementation:**

```ts
import { test, expect } from "@playwright/test";

/**
 * D-04 + D-06 regression net: phones (<768px) never fetch a video file.
 * Even though the SSR HTML still contains the <video> element for a
 * hydration tick, every <source media> query starts with
 * "(min-width: 768px) and ...", so a 412px phone matches zero sources
 * and the browser starts zero downloads.
 *
 * If any future change re-opens a vector for mobile video fetch
 * (new <source> without bounded media, <link rel=preload as=video>,
 * service worker prefetch, etc.), this spec fails.
 */
test.describe("HERO-04: mobile is video-free", () => {
  test("412×823 viewport fires zero video network requests", async ({ page }) => {
    const videoUrlPattern = /\.(mp4|webm|m4v|mov)(\?|$)/i;
    const videoRequests: string[] = [];

    page.on("request", (req) => {
      const url = req.url();
      if (videoUrlPattern.test(url)) {
        videoRequests.push(url);
      }
    });

    // Match LHCI mobile profile exactly: 412×823 (lighthouserc.json
    // screenEmulation block, verified at HEAD).
    await page.setViewportSize({ width: 412, height: 823 });
    await page.goto("/");
    await page.waitForLoadState("networkidle");

    // Scroll the page to surface any lazy-loaded video. The hero is
    // GSAP-pinned on desktop but renders a static stack on mobile per
    // M4 Phase 3, so scrolling shouldn't trigger any video fetch. This
    // is belt-and-braces: future regressions where someone adds a
    // mobile-specific scroll-triggered video would also fail here.
    await page.evaluate(() => window.scrollTo(0, document.body.scrollHeight));
    await page.waitForTimeout(500);
    await page.evaluate(() => window.scrollTo(0, 0));
    await page.waitForTimeout(500);

    expect(
      videoRequests,
      `expected zero video requests at 412×823 (D-04 mobile-video-free guarantee), got: ${JSON.stringify(videoRequests)}`
    ).toHaveLength(0);
  });
});
```

**Why this shape:**
- Filename matches the directory's plain-noun convention (`platform-mobile`, `touch-targets`, `reduced-motion`, etc.).
- Viewport 412×823 matches the LHCI mobile profile exactly (lighthouserc.json screenEmulation block — verified). The Playwright spec catches anything LHCI would catch, before LHCI runs.
- Regex `/\.(mp4|webm|m4v|mov)(\?|$)/i` catches the canonical video extensions. Case-insensitive for safety. The `(\?|$)` boundary prevents false positives on substrings.
- `waitForLoadState("networkidle")` lets the page settle. The scroll-then-wait pattern is defense against future lazy-load scenarios.

```bash
#!/usr/bin/env bash
# Asset size budget guard for public/hero/.
# Exits 0 if every asset is under its budget; non-zero (with diagnostic)
# if any asset blows past.
#
# Budgets are derived from Phase 5.1 D-07 (CONTEXT.md):
# current shipping size + ~10% headroom, rounded up.

set -euo pipefail

DIR="public/hero"

# Budgets in bytes (10 MB = 10_485_760; 6 MB = 6_291_456; 3 MB = 3_145_728;
# 200 KB = 204_800; 300 KB = 307_200). Using base-1024 ("binary" MB/KB)
# because that's what `stat` reports.
declare -a BUDGETS=(
  "homepage-hero-720p.mp4:10485760"
  "homepage-hero-540p.mp4:6291456"
  "homepage-hero-360p.mp4:3145728"
)

FAIL=0
for entry in "${BUDGETS[@]}"; do
  FILE="${entry%%:*}"
  BUDGET="${entry##*:}"
  PATH_FULL="$DIR/$FILE"

  if [ ! -f "$PATH_FULL" ]; then
    echo "MISSING: $PATH_FULL" >&2
    FAIL=1
    continue
  fi

  # macOS `stat -f %z`, Linux `stat -c %s`. Use a portable fallback.
  if SIZE=$(stat -f %z "$PATH_FULL" 2>/dev/null); then
    :
  else
    SIZE=$(stat -c %s "$PATH_FULL")
  fi

  if [ "$SIZE" -gt "$BUDGET" ]; then
    echo "FAIL  $FILE: $SIZE bytes > $BUDGET bytes budget" >&2
    FAIL=1
  else
    PCT=$(( SIZE * 100 / BUDGET ))
    echo "PASS  $FILE: $SIZE bytes ($PCT% of budget)"
  fi
done

exit "$FAIL"
```

  - The 3 MP4 budgets cover the actual regression scenario (encoder produces files larger than expected). The 3-budget script is sufficient.
- If the planner disagrees and wants 5 budgets, the script extends trivially. But: this researcher recommends 3.

**CI integration:** Add a step to `.github/workflows/perf.yml` (currently a 30-line file that runs lhci autorun on `main` and PRs). Insert before the lighthouse step:

```yaml

```
