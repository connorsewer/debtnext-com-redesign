## WebM strategy

| Option | Description | Selected |
|--------|-------------|----------|
| Drop WebM entirely (Recommended) | Ship MP4 only. Every browser plays it fine. Smallest, fastest, lowest-risk fix. The MP4 ladder alone meets Phase 5's speed goal. | ✓ |
| Loosen WebM keyframes | Keep WebM but stop marking every frame. Files get small, but if browsers pick WebM, scroll-scrubbing may stutter between keyframes. | |
| Switch WebM to AV1 | Newer codec, better at our use case in theory. But longer encode times, untested scrub quality, and adds build complexity. | |

**User's choice:** Drop WebM entirely (Recommended)
**Notes:** Locked as **D-01**. Investigation during context gathering uncovered that the preservation summary's "CRF-only without 2-pass" diagnosis was wrong — the script already does VP9 2-pass with `-b:v` targets. The real cause is structural: per-frame keyframes (locked by D-05 from Phase 5 for scroll-scrub fidelity) are incompatible with VP9 bitrate targeting because every frame becomes a full intra-coded picture. Dropping WebM bypasses the fundamental issue rather than trying to tune around it.

## Mobile gate fix

| Option | Description | Selected |
|--------|-------------|----------|
| Tighten HTML source rules (Recommended) | Change source rules so phones match zero sources. Simplest, fully effective, no CDN complexity. | ✓ |
| Server-side phone detection | Skip the video element entirely on phones at server-render time. Most 'architecturally correct' but splits CDN caching. | |
| Both | Server-side detection + tightened source rules. Most defensive. Adds CDN caching complexity. | |

**User's choice:** Tighten HTML source rules (Recommended)
**Notes:** Locked as **D-04**. Investigation revealed the real defect mechanism is two-fold: (a) SSR renders `<video>` element because `isMobile` defaults to `false` on the server, and (b) the bounded `<source media>` queries use unbounded lower edges (`(max-width: 1023px)`) so a 412px phone matches the 360p source. Tightening the queries to bounded ranges (`(min-width: 768px) and (max-width: 1023px)`) makes mobile match zero sources, so even if the element exists for a hydration tick, no download starts. Server-side detection was rejected as redundant given the source-rule fix is provably watertight; deferred to a future phase if a second use case emerges.

## Regression test design

| Option | Description | Selected |
|--------|-------------|----------|
| Both (Recommended) | Network watcher (no video files requested at phone size) + file-size budget (each hero file capped). | ✓ |
| Network watcher only | Catches mobile-downloading-video. Doesn't catch encoder regressions where files balloon but stay functional. | |
| File-size budget only | Catches encoder regressions. Doesn't catch mobile-gate regressions. | |

**User's choice:** Both (Recommended)
**Notes:** Locked as **D-06** (network watcher) + **D-07** (size budget). Each guards a different failure mode — the network watcher catches mobile-gate regressions, the size budget catches encoder regressions. Both are cheap (~30 lines and ~20 lines respectively) and pay for themselves the first time either catches a regression. Specific size budgets set at "current shipping size + ~10% headroom" — exact numbers in CONTEXT.md.
