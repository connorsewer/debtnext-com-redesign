# Phase 05.1: HERO-04 gap closure — WebM encoder re-tune and mobile video gate — Context

**Gathered:** 2026-05-21
**Status:** Ready for planning

<domain>
## Phase Boundary

Close the gap that left Phase 5 PARTIAL. Phase 5 shipped HERO-01..03 but Plan 05-05 (the LHCI ship gate) failed at Case C — representative-run `/` LCP measured 16,219 ms against the 2,300 ms gate (over budget, not flake). Phase 5.1 fixes the two defects that caused the failure and re-runs the gate so Phase 5 can close.

The two defects:

1. **Mobile is not actually video-free.** Despite D-04 from Phase 5 ("mobile remains video-free"), Lighthouse mobile (412×823) downloads `homepage-hero-360p.webm` (8.88 MB). Two reasons: (a) `<video>` element is in the server-rendered HTML (`isMobile` defaults to `false` on the server) so the browser starts downloading before React's `matchMedia` check unmounts it; (b) the `<source media>` queries themselves use `(max-width: 1023px)` for the 360p tier — a 412px phone matches that query, so the browser picks the 360p source as soon as it sees the element.
2. **WebM ladder is bigger than the MP4 ladder.** Every WebM tier exceeds its MP4 counterpart — worst case 360p WebM is 4.47× its MP4. The build script already does VP9 2-pass with `-b:v` targets (1500k/900k/500k), so the preservation summary's "CRF-only without 2-pass" hypothesis is wrong. The real cause: per-frame keyframes (D-05) are fundamentally incompatible with VP9 bitrate targeting. All-intra VP9 cannot meaningfully compress; every frame is a full keyframe.

In-scope for Phase 5.1:
- Drop WebM entirely from the ladder (cause is structural, not tunable)
- Tighten mobile gate so phones literally match zero `<source>` queries
- Add regression guards (mobile network watcher + hero asset size budgets)
- Fold Phase 5's parked 05-05 closing tasks (gauge re-run, manual walkthrough, docs writeup, traceability flips) into Phase 5.1's closing wave so Phase 5 closes when 5.1 closes

Out of scope (carried from Phase 5, locked, not re-discussed):
- Ladder shape (720p / 540p / 360p — no new tiers)
- MP4 ladder file targets (Phase 5 already produced acceptable sizes: 9.3 MB / 4.6 MB / 1.9 MB)
- LHCI throttling profile (mobile, 412×823, DPR 1.75, `cpuSlowdownMultiplier=4`, 3 runs, median assertion at 2300 ms)
- AVIF poster (49 KB at 1920w, preload + fetchPriority="high" — HERO-03 shipped clean)
- General Sans self-host (HERO-02 shipped clean)
- AV1 codec exploration (deferred — Phase 5.1 ships MP4-only; AV1 is a future investigation)
- Server-side User-Agent phone detection (deferred — source-rule fix + Playwright guard is sufficient)
- Mobile cinematic playback (M4 Phase 3 mobile static-stack architecture stands)

<decisions>
## Implementation Decisions

### WebM strategy (defect B)

- **D-01:** **Drop WebM entirely from the hero ladder.** Ship MP4-only at all three tiers (720p / 540p / 360p). Every browser plays MP4 cleanly; Chrome/Firefox/Edge lose a small theoretical efficiency they were never actually getting from the oversized WebM files. Rationale: VP9 with per-frame keyframes (locked by D-05 from Phase 5) cannot hit any meaningful bitrate target — every frame is a full intra-coded picture. Fixing the encoder while keeping WebM requires either dropping per-frame keyframes on WebM (degrades scroll-scrub on WebM-picking browsers) or switching to AV1 (slow encoding, untested scrub quality). Dropping WebM is the smallest blast radius that meets the perf gate.

- **D-02:** **Full cleanup, not partial.** `scripts/build-hero-ladder.sh` drops its `libvpx-vp9` encode block entirely; the script produces only MP4 variants. The six existing WebM files in `public/hero/` are deleted in the same commit as the script change. Rationale: leaving dead files in `public/hero/` invites drift and confuses future readers. Git history preserves the prior WebM build path if AV1 exploration ever wants the precedent.

- **D-03:** `src/content/homepage-hero.ts` `heroCinematic.media.video` sources array drops the three WebM entries; only the three MP4 sources remain. `HomepageHero.tsx` doesn't change — it already maps the sources array generically.

### Mobile gate fix (defect A)

- **D-04:** **Tighten the `<source media>` queries so phones (<768px) match zero sources.** Specifically: the 360p source becomes `(min-width: 768px) and (max-width: 1023px)` (was `(max-width: 1023px)`); the 540p source becomes `(min-width: 768px) and (max-width: 1439px)` (was `(max-width: 1439px)`); the 720p source remains unbounded but applies only above 1440px implicitly because narrower viewports match the bounded sources first. A 412px phone now matches no source; even if the `<video>` element exists in the SSR HTML for a hydration tick, the browser starts no download because no source qualifies.

- **D-05:** **No server-side User-Agent phone detection in 5.1.** The `<source media>` fix is provably watertight (no matching source = no download, full stop). Adding server-side phone detection would split the CDN cache into mobile and desktop variants — extra complexity for redundant coverage. Re-evaluate only if a second use case for mobile-aware SSR emerges. The `!isMobile` React gate stays in place as belt-and-suspenders inside the client tree.

### Regression guards

- **D-06:** **Add a mobile video-free Playwright spec.** New file: `tests/responsive/hero-mobile-video-free.spec.ts`. Opens `/` at 412×823 (matches LHCI mobile profile + D-07 from Phase 5), intercepts every network request, asserts zero requests with URLs matching `/\.(mp4|webm|m4v|mov)$/` for the full page lifecycle. Fails the test (and CI) if any video request fires. This is the direct regression net for defect A.

- **D-07:** **Add a hero asset size budget guard.** New script (or step in the existing CI workflow): walk `public/hero/` and assert each file is under its budget. Budgets (current size + ~10% headroom, rounded up for slack):
  - `homepage-hero-720p.mp4` ≤ 10 MB
  - `homepage-hero-540p.mp4` ≤ 6 MB
  - `homepage-hero-360p.mp4` ≤ 3 MB
  - `homepage-hero-start.avif` ≤ 200 KB (HERO-03 contract)
  - `homepage-hero-start.png` ≤ 300 KB (WebP/PNG fallback if still shipping)
  Fails the build if any asset blows past its budget. This is the direct regression net for defect B.

### Phase 5 closing wave (fold-in)

- **D-08:** **Phase 5.1's closing wave absorbs Phase 5's parked 05-05 tasks.** The four parked tasks (LHCI gauge re-run, manual walkthrough at 1440/1024/768 viewports, write `docs/m5-phase-5-lhci-run.md`, flip HANDOFF.md + .planning/PROJECT.md + .planning/STATE.md + .planning/REQUIREMENTS.md to shipped) execute as Phase 5.1's final wave. When Phase 5.1 ships, Phase 5 ships simultaneously. The traceability table flips HERO-01..04 to Done in the same commit. The `05-05-PLAN.md` file remains in place as a historical record of the parked work but its status flips to "absorbed by 05.1."
