<decisions>
## Implementation Decisions (LOCKED)

### Lazy-load strategy
- GSAP, `gsap/ScrollTrigger`, and `@gsap/react`'s `useGSAP` must be loaded via dynamic
  `import()` **inside** the desktop, non-reduced-motion code path only — i.e. behind the
  existing `!isMobile && !prefers-reduced-motion` gate. They must NOT appear as top-level
  `import` statements in any module reachable from the `/` Server Component tree's eager
  client bundle.
- Both eager importers must be converted: `src/components/sections/HomepageHero.tsx` AND
  `src/components/sections/HomepageHandoffSection.tsx`. The handoff section's ScrollTrigger
  is coupled to the hero's pin spacer (see `HomepageHandoffSection.tsx:80-82`,
  `ScrollTrigger.refresh()`), so the two must stay coordinated — load order and
  `registerPlugin` must not double-register or race.
- `gsap.registerPlugin(ScrollTrigger, useGSAP)` currently runs at module top level in both
  files; it must move inside the lazy path so it runs only after the dynamic import resolves.
  Keep it to a single call site; `registerPlugin` is idempotent, so per-section controller
  mounts are a harmless no-op (no double-register, no race).

### Behavior parity (non-negotiable)
- Desktop cinematic (pin, scrub, crossfades, framed-dashboard handoff) is byte-for-byte
  unchanged in observable behavior.
- Mobile (`<768px`) and `prefers-reduced-motion` already skip the cinematic; after this
  change they must additionally never download GSAP.
- Fail-open: if the dynamic import rejects, the page must render the static start-frame +
  overlay (current mobile fallback), never a broken/blank hero.

### Verification
- Add `tests/responsive/hero-gsap-free-mobile.spec.ts` (model on the existing
  `tests/responsive/hero-mobile-video-free.spec.ts` 412×823 network-watcher): assert zero
  requests whose URL matches `gsap` are made on a mobile viewport load.
- All existing regression nets stay green (`hero-mobile-video-free`, `source-ladder`,
  `poster-avif-negotiation`, `check-hero-assets.sh`, full Playwright suite).
- LHCI Case C re-run is the gate. Per `project_next_server_hangs_in_sandbox`: do NOT rely
  on a local `next build/start`; verify via Vercel preview deploy + LHCI logs.

### D-08 close-out (only after gate is green)
- Single doc-sync: HERO-01..04 → Done; STATE.md M5 resume context closed; ROADMAP.md
  evolution note; HANDOFF.md. Per `feedback_docs_in_sync`, docs land in the same commit
  series as the code.
- If the gate still misses after dynamic-import: STOP and escalate to the user with the
  measured number and the fallback menu (relax gate to 2,500 ms §12 floor, etc.). Do not
  silently relax the gate.

<specifics>
## Specific Ideas

- Stack: Next 16.2.6, React 19.2.4, gsap ^3.15.0, @gsap/react ^2.1.2, framer-motion ^12.39.0.
- `useGSAP` is GSAP's React hook; under dynamic import, the cleanest pattern is to gate the
  whole cinematic effect so the `import()` resolves first, then run the GSAP setup inside an
  effect (the hook's scope/cleanup contract must be preserved). The planner should decide
  between (a) dynamically importing inside a `useEffect` and driving ScrollTrigger imperatively,
  vs (b) a `next/dynamic` client subcomponent that owns all GSAP and is only mounted on
  desktop — and document the choice. Option (b) keeps GSAP out of the parent chunk most
  reliably; verify with the bundle/network assertion either way.
- There is a `useIsMobile` hook already in use by the handoff section.
</specifics>
