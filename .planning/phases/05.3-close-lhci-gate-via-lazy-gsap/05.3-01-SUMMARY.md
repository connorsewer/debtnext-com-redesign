# Phase 05.3 Plan 01: Lazy-load GSAP off the homepage critical path Summary

Moved GSAP, ScrollTrigger, and `@gsap/react`/`useGSAP` out of the `/` eager client chunk by relocating all GSAP into a single `next/dynamic({ ssr:false })` desktop-only `HeroCinematicController` that `await import()`s the libraries inside an effect; the two homepage sections now carry zero top-level GSAP imports and mount the controller only behind the `!isMobile && !prefersReducedMotion` gate, so mobile and reduced-motion sessions never download GSAP.

## What shipped

- **`HeroCinematicController.tsx` (new, ~225 lines)** — the only module in `src/` that references GSAP after this plan. Renders `null`, owns no DOM. Inside one `useEffect([])`: `await import("gsap")` / `import("gsap/ScrollTrigger")` / `import("@gsap/react")` (wrapped in try/catch for fail-open), then `gsap.registerPlugin(ScrollTrigger, useGSAP)` once. It builds the hero master ScrollTrigger (pin/scrub/ease windows/crossfades) and the handoff tab-progression ScrollTrigger (`vhPerTab`/`tabCount`/`Math.floor`) from refs/callbacks supplied by the parents, runs `ScrollTrigger.refresh()`, and on unmount removes the `loadedmetadata` listener and kills every trigger it created.
- **`HomepageHero.tsx`** — removed the 3 eager GSAP imports + top-level `registerPlugin` + the entire `useGSAP` block. Added `next/dynamic` import, a `prefersReducedMotion` matchMedia effect, and `cinematicEnabled = !isMobile && !prefersReducedMotion`. Mounts `<HeroCinematicController heroRefs={...} />` only when `cinematicEnabled`. All refs, the form handler, and the static start-frame + overlay (the fail-open render) are unchanged.
- **`HomepageHandoffSection.tsx`** — removed the 3 eager GSAP imports + `registerPlugin` + the `useGSAP` block. Moved `activeIdRef` above the new `onActiveTab(idx)` callback (which runs the identical guarded `setActiveId` update), added `prefersReducedMotion`, and mounts `<HeroCinematicController handoff={{ sectionRef, tabCount, vhPerTab: VH_PER_TAB, onActiveTab }} />` behind the same gate. Mobile static tree, tab state, `scrollToTab`, and keyboard handlers unchanged.
- **`tests/responsive/hero-gsap-free-mobile.spec.ts` (new)** — 412x823 network-watcher (modeled on `hero-mobile-video-free.spec.ts`) asserting zero requests matching `/gsap/i` after load + full scroll + scroll-back; `toHaveLength(0)` with a `JSON.stringify` failure message.

## Behavior parity

The hero scrub math and the handoff progression math were lifted verbatim. No numeric coefficient, easing window (`ease(0,0.12)`, `ease(0.0,0.03)`, `ease(0.7,0.88)`, `ease(0.78,0.88)`, `ease(0.70,0.88)`, `ease(0.95,1.0)`), selector (`[data-handoff-section]`), pin/scrub setting (`pin: sticky`, `pinSpacing: true`, `start: "top top"`, `end: +=innerHeight*2.6`, `scrub: 0.5`), or scroll length (`innerHeight * VH_PER_TAB * tabCount`, `VH_PER_TAB = 0.75`) was changed. The deferred `loadedmetadata` wire path and the live matchMedia re-checks inside `wire()` are preserved.

## Mounting strategy (decision)

**Per-section controller instances**, not a single shared instance. The hero mounts one controller with `heroRefs`; the handoff mounts a second with `handoff` wiring. `gsap.registerPlugin` is idempotent, so the second mount's registration is a harmless no-op (no double-register, no race). Both controllers call `ScrollTrigger.refresh()` after creating their trigger, and the handoff trigger keeps `invalidateOnRefresh: true`, which preserves today's belt-and-suspenders pin-spacer refresh ordering. The plan's fallback (single shared instance owned by the hero) was not needed; if the preview behavior check surfaces pin-spacer ordering flakiness, that is the documented fallback for Plan 02.
