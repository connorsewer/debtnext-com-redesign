<objective>
Remove eagerly-imported GSAP + ScrollTrigger + useGSAP from the homepage's critical
JS path so the LHCI Case C simulator stops modeling ~80-120 KB of GSAP against the `/`
CPU/JS graph. GSAP must load only via dynamic `import()` inside the desktop,
non-reduced-motion code path, in a single owner module, with `registerPlugin` called idempotently (never double-registering even if both sections mount their own controller).

Purpose: HERO-04's residual LHCI gap (3,869 ms vs 2,300 ms gate) is the simulator's
projection of the JS critical path + CPU graph, not an asset fetch (observed LCP is
1,254 ms). Dynamic-importing GSAP is the locked, highest-expected-win lever
(per docs/m5-phase-5-lhci-run.md "Recommended next move", option 1).

Output: A shared HeroCinematicController module, two refactored section components
with zero top-level GSAP imports, and a regression spec proving GSAP is never fetched
on mobile. Desktop cinematic behavior is byte-for-byte unchanged.
</objective>
