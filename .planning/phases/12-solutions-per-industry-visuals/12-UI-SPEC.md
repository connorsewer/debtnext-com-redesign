## Typography

Source: DESIGN.md §9.1 fluid type scale (Utopia clamp tokens, 360px/1440px anchors). This phase adds zero type sizes. The archetype components own all type styling; payloads supply strings only. The subset this phase's surfaces actually render:

| Role | Size (token) | Weight | Line Height |
|------|--------------|--------|-------------|
| Visual caption / eyebrow / pill text | 12px (`--text-caption`, static) | 420 (UI emphasis) | 1.3 |
| Console row labels, schematic node labels, card subs | 13→14px (`--text-body-sm`) | 400 | 1.5 |
| Body adjacent to visuals, annotation text | 15→16px (`--text-body-md`) | 400 | 1.5 |
| KPI values / card headline numbers | 18→21px (`--text-body-lg` family, per archetype internals) | 480 (display emphasis), `tabular-nums` | 1.2 |

Rules bound for this phase:
- Weight changes are sparing: 400 standard, 420 UI emphasis, 480 display emphasis (DESIGN.md line 385). Never bold-stack inside a visual.
- All animated or live numeric values keep `tabular-nums` (no width jitter — `LiveValue` primitive already enforces this).
- Sentence case for every string in every payload: titles, pills, tags, annotations, ariaSummaries (CLAUDE.md §5).
- One H1 per page is untouched — visuals contribute no headings to the document outline (they are `role="img"`).

| Role | Value | Usage |
|------|-------|-------|
| Dominant (60%) | `#171721` (`--background` / `--product-canvas`) | Page shell and the canvas inside every archetype frame |
| Secondary (30%) | `#20212d` / `#1e1e2a` (`--card` family) | Elevated surfaces: console row chrome, schematic node fills, hub cards |
| Accent (10%) | `#5266EB` (`--primary` / `--chart-1`) | Filled "Request a demo" CTAs, primary chart series, focus anchors ONLY |

Accent reserved for: the one filled "Request a demo" pill per band, the primary data series in charts, selected-tab fills, and focus anchors. Never as text on the dark canvas (3.77:1, fails AA — use `--accent-text-dark #7280F4` for accent text). Never on multiple competing elements in one band.

### Industry accent mapping (D-11, AUDIT P12-02 — binding)

No new colors. The deleted widget's invented hexes (`#4AA8C9`, `#8472F0`, `#3D9DE0`) die with it. Industry accents come exclusively from the existing chart tokens. `--chart-2` (`#323649`) is near-canvas and is reserved for baselines/grids/muted comparison — it is NOT a usable accent. That leaves 4 usable hues for 6 industries, so accents repeat by design; distinctness rides composition + data, not hue (D-08).

| Industry | Accent token | Hex | Note |
|---|---|---|---|
| Utilities | `--chart-5` | `#06b6d4` | nearest-hue successor to the dying #4AA8C9 |
| Financial services | `--chart-1` | `#5266EB` | unchanged from the widget |
| Telecom | `--chart-3` | `#10b981` | no token equivalent for the dying violet; composition differentiates from insurance |
| Fintech | `--chart-5` | `#06b6d4` | shares hue with utilities — compositions must differ visibly (map-enforced) |
| Insurance | `--chart-3` | `#10b981` | shares hue with telecom — compositions must differ visibly (map-enforced) |
| Healthcare | `--chart-4` | `#f59e0b` | |
| Hub (cross-industry) | per-card, all 4 usable tokens | — | cards pass `accent: "var(--chart-N)"` strings into the cards-branch `color-mix()` styles |

The planner may adjust this mapping inside `12-ARCHETYPE-MAP.md` provided: (a) only `chart-1/3/4/5` are used, (b) any two industries sharing an accent differ in archetype composition on every co-visible surface, and (c) the map's distinctness table records the check.

### Status color contract (D-12, Pitfall 8 — binding)

Every status color is paired with a text label or icon, with no exceptions (DESIGN.md line 348). `--chart-3` green = positive, `--chart-4` amber = warning/pending, `--chart-5` cyan = informational, `--chart-1` indigo = primary series. Every data story must survive grayscale: if the page is rendered without color, the industry's story must still be fully legible from labels, values, and composition.

## Copywriting Contract

All payload strings are governed copy under CLAUDE.md §5. The largest new-copy surface this phase is ~25 payloads × several strings each (titles, subs, pills, tags, annotations, ariaSummaries).

| Element | Copy |
|---------|------|
| Primary CTA | "Request a demo" — exact label, one filled instance per band, untouched by this phase. Visuals never carry CTAs and never visually compete with the band's CTA. |
| Empty state heading | not applicable — payloads are statically typed and always present; an "empty" archetype is a build error, not a runtime state |
| Empty state body | not applicable (same reason) |
| Loading state | Lazy skeleton: `VisualSkeleton` pulse on `--product-canvas`, `rounded-[16px]`, height matched to the resolved visual (see Interaction States below) — no text, no spinner |
| Error state | Fail-open contract (FND-01): if a reveal or lazy load fails, content renders fully visible in final state; nothing is ever stuck at `opacity: 0`. No error copy exists because no error UI exists. |
| Destructive confirmation | not applicable — no destructive actions on a marketing site |

### Payload copy rules (binding, per-module self-check)

1. No em dashes anywhere. The established console separator is `·` (e.g. "3 vendors · 30-day").
2. No "not X, it's Y" constructions; state the positive claim directly.
3. No banned phrases (seamless, leverage, robust, unlock, elevate, empower, etc. — full list CLAUDE.md §5).
4. Sentence case, contractions, digits for numbers.
5. Industry-specific nouns are the differentiator and the spec's distinctness hook: arrears/deposits (utilities), charge-off/Reg F (financial services), prepaid/postpaid (telecom), BNPL/installments (fintech), subrogation/earned premium (insurance), self-pay/balance-after-insurance/EBO (healthcare).
6. Vendor/TSI framing stays agency-network-agnostic ("routes across the originator's existing vendors"); never independence claims (CLAUDE.md §6).
7. Every payload module carries `[CLAIMS REVIEW]` / `[COI REVIEW]` header tags — audit-trail markers, non-blocking per Andrew Budish's 2026-06-12 sign-off (D-09).
8. Figures are real-shaped, anonymized, generic; plausible relative magnitudes across industries (telecom counts > insurance counts; utilities balances < financial-services balances). No named clients, no invented testimonials (CLAUDE.md §15).
9. `ariaSummary` is mandatory on every archetype instance and is full governed prose that tells the industry's story in words (see Accessibility below).
10. `visualLabel` fields in `src/content/solutions-*.ts` must be updated to accurately describe the new visuals when wired (they feed the motion wrapper's `aria-label`).

## Interaction States

This phase ships static archetype rendering (D-04 — Explorable shells not required; zero-cost discretion only). The 7-state component matrix from CLAUDE.md §10 applies to the archetypes as already built; the phase-specific contracts are:

| State | Contract |
|-------|----------|
| Default | Archetype renders fully from its typed payload; all values visible without interaction (default-visible values, asserted by `solutions-visuals.spec.ts`) |
| Loading | Lazy `ssr:false` skeleton. **Accordion visuals:** shared 20rem `VisualSkeleton` (unchanged, D-14). **Console heroes in `ProductVisualBand`:** the bare band has no height floor and the 20rem skeleton will undershoot a hero-grade Console (header KPI + 4 rows + pills + callout resolves taller — Pitfall 1 / research A2). Contract: author all 6 hero payloads to one consistent row/pill budget so all 6 heroes resolve to the same box; measure the resolved height on the preview; then reserve that exact box following the `FLAGSHIP_SKELETON_MIN_H` precedent in `lazy.tsx` (dedicated taller skeleton constant or a `min-h` wrapper on the band child). The skeleton must match the resolved box, not approximate it. Verified by the `/solutions/utilities` LHCI CLS gate (< 0.1). |
| Hover / Focus | Inherited from archetype internals (CSS-engine ambient/hover per DESIGN.md §4.7); no new hover behavior is added this phase. Focus-visible: `#9CB4E8` 2px outline, 2px offset, anywhere a focusable element exists (accordion triggers — already compliant). |
| Active | Accordion item activation cross-fades visuals via the proven `FeatureAccordion` mechanism; no new transitions |
| Disabled | not applicable (no interactive controls added) |
| Error | Fail-open: final visible state, never stuck `opacity: 0` (guarded by the existing reveal-fail-open spec) |
| Reduced motion | **Full data parity, zero motion** (D-05 contract from Phase 11): every value, label, and composition a motion-enabled user sees is present and final for a `prefers-reduced-motion: reduce` user. `LiveValue` lands on the displayed value immediately. Asserted per route by `solutions-visuals.spec.ts` (port `assertReducedMotionDataParity` + `assertNoStuckOpacity` with the `iterations === Infinity` idle-guard). |

### Motion contract

Engine-per-job (DESIGN.md §4.7): all motion in this phase comes from the motion barrel via the archetypes — Framer for entrances/reveals (Reveal), live values (LiveValue), CSS for ambient/hover. No GSAP (no scroll-scrub/pin surfaces in scope). Pages never import Framer/GSAP directly; pages import only `ConsoleVisual` / `DataStoryVisual` / `SchematicVisual` from `archetypes.tsx` plus payload modules. Never import `Console.tsx` / `DataStory.tsx` / `Schematic.tsx` directly (Pitfall 5 — eager-import TBT blowout; the 300ms content-route bar has limited headroom).

## Accessibility Contract

- Every archetype instance carries `role="img"` + a complete `ariaSummary` (D-12, Phase 11 pattern). The ariaSummary is the screen-reader equivalent of the visual's full story, written in governed prose, industry-specific (it doubles as a distinctness assertion hook in the spec).
- Status colors always label-paired (text or icon beside color) — the grayscale-survivability test is the bar.
- WCAG 2.2 AA contrast on every pairing; body text on dark uses `#EDEDF3` / `#C3C3CC` hierarchy; `--primary` never as text on dark.
- 44px touch floor on any interactive element (accordion triggers already compliant; this phase adds no new interactive elements).
- One H1 per page unchanged; visuals contribute nothing to the heading outline.
- Existing axe + touch-target specs already cover all solutions routes (AUDIT FIX-02); they must stay green.

## Distinctness Contract (SOLVIS-01 — the reason this phase exists)

This is a verifiable gate, not a vibe (CONTEXT specifics + Pitfall 2):

1. **Map gate first:** `12-ARCHETYPE-MAP.md` (D-08) is produced before any page work and carries an explicit distinctness table — per-item one-line data-shape intents, no two industries alike in composition. Special pressure: utilities/insurance/healthcare share near-identical "Reconcile daily" reporting items and parallel "configure X and Y separately" optimization items; the payloads must differentiate them with different row nouns, different chart kinds where possible, and different numbers.
2. **Schematic variation:** with all 6 `placement` items likely mapped to Schematics, node/edge composition must vary per industry — different source nodes (billing/CIS, core banking, OSS/BSS, lending API, policy admin, EHR/clearinghouse) and different pool structures.
3. **Spec gate second:** `solutions-visuals.spec.ts` asserts per-route industry-unique strings (account-type nouns in Console values and ariaSummary fragments), on load AND under reduced motion. "role=img exists" alone is insufficient — it cannot distinguish two industries that both render archetypes, and it cannot catch a copy-pasted payload (Pitfall 4).
4. **ID-collision guard:** the 18 accordion item ids (`placement`/`optimization`/`issues`/`reporting`) collide with the homepage `VISUALS` registry; the page-supplied `visuals` prop takes precedence (verified, `FeatureAccordion.tsx:159-162`). Do NOT rename item ids (breaks `#feat-{id}-button` anchors and `accordion_toggle` analytics).
